# 모르는 것을 물어볼 때, 정확한 답변 얻는 법

AI는 자신 있게 틀린 답(hallucination)을 줄 수 있다. 핵심은 **AI가 틀릴 여지를 줄이는 질문 습관**을 만드는 것이다.

---

## 1. "모른다"고 말할 수 있게 허용하라

AI는 기본적으로 답을 만들어내려는 경향이 있다. 명시적으로 불확실성을 표현할 여지를 줘야 한다.

**나쁜 예:**
> Go 1.23에서 추가된 iter 패키지의 사용법을 알려줘

**좋은 예:**
> Go 1.23에서 iter 패키지가 추가됐다고 들었는데, 정확히 알고 있으면 사용법을 알려줘. 확실하지 않으면 모른다고 말해줘.

---

## 2. 검증 가능한 형태로 답을 요청하라

출처나 근거를 함께 요청하면 틀린 답을 걸러내기 쉬워진다.

- "공식 문서 기준으로 설명해줘"
- "이 정보의 출처가 뭐야?"
- "코드로 보여주고, 실행 결과도 예측해줘"

답변에 출처가 모호하거나, "일반적으로 ~라고 알려져 있다" 식의 표현이 많으면 의심해봐야 한다.

---

## 3. 큰 질문을 쪼개서 물어라

복잡한 질문 하나보다 단순한 질문 여러 개가 정확도가 높다.

**나쁜 예:**
> Kubernetes에서 HPA와 VPA를 함께 쓸 때 주의할 점과 설정 방법을 알려줘

**좋은 예:**
> 1. Kubernetes HPA는 어떤 메트릭 기준으로 스케일링해?
> 2. VPA는 어떤 경우에 쓰는 게 적절해?
> 3. HPA와 VPA를 동시에 쓸 때 충돌이 발생할 수 있어?

단계별로 물으면 어느 지점에서 AI의 답이 부정확한지 파악하기도 쉽다.

---

## 4. 반론을 요청하라

AI는 사용자의 의견에 동조하는 경향이 있다. 의도적으로 반대 관점을 요청하면 더 균형 잡힌 답을 얻을 수 있다.

- "이 접근의 단점은 뭐야?"
- "반대 의견을 가진 사람은 뭐라고 주장할까?"
- "이게 틀릴 수 있는 경우가 있어?"

---

## 5. AI의 답을 교차 검증하라

AI 답변을 최종 결론이 아닌 **출발점**으로 활용하라.

| 방법 | 설명 |
|------|------|
| 같은 질문을 다르게 물어보기 | 표현을 바꿔서 물어봤을 때 답이 달라지면 불확실한 영역 |
| 공식 문서와 대조 | AI가 알려준 API나 설정을 공식 문서에서 확인 |
| 직접 실행 | 코드나 명령어는 반드시 직접 돌려보기 |
| 다른 모델에게 검증 | 같은 질문을 다른 AI 모델에게도 물어보기 |

---

## 6. 컨텍스트를 충분히 제공하라

AI는 주어진 정보 안에서 답을 만든다. 맥락이 부족하면 AI가 알아서 채우게 되고, 그게 hallucination의 원인이 된다.

**포함하면 좋은 것들:**
- 사용 중인 언어, 프레임워크, 버전
- 현재 상황과 시도한 것
- 원하는 결과와 제약 조건

---

## 7. 최신 정보는 AI를 믿지 마라

AI 모델에는 학습 데이터의 마감일(knowledge cutoff)이 있다. 최신 버전, 최근 뉴스, 변경된 API 등은 틀릴 가능성이 높다.

**대안:**
- 웹 검색 기능이 있는 AI를 사용하거나
- AI에게 "이건 최신 정보가 필요한 질문이야. 네 학습 데이터 기준으로 답하고, 변경됐을 수 있다고 표시해줘"라고 요청

---

## 요약

| 원칙 | 핵심 |
|------|------|
| 모른다고 말할 여지를 줘라 | "확실하지 않으면 모른다고 해줘" |
| 검증 가능한 답을 요청하라 | 출처, 코드, 공식 문서 기준 |
| 질문을 쪼개라 | 큰 질문 하나보다 작은 질문 여러 개 |
| 반론을 요청하라 | 단점, 반대 의견, 틀릴 수 있는 경우 |
| 교차 검증하라 | AI 답변은 출발점이지 결론이 아니다 |
| 컨텍스트를 줘라 | 맥락이 부족하면 AI가 지어낸다 |
| 최신 정보는 의심하라 | knowledge cutoff 이후 정보는 틀릴 수 있다 |
